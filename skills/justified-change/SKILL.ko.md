# Skill: Justified Change (한국어 페어)

> 영문 기본 문서: `SKILL.md`

## 목적

코드 변경을 요청된 목표에 비례하고, 근거가 있으며, 검증 가능한 상태로 유지한다.

이 스킬의 목적은 변경을 무조건 작게 만드는 것이 아니다.
목표를 만족하기 위해 필요한 일은 필요한 만큼 수행하되, 변경 범위는 설명 가능해야 한다.

## 사용할 때 / 쓰지 않을 때

- 비자명한 구현, 버그 수정, 리팩터링, 코드 리뷰 반영 전에 쓴다.
- 비자명한 코드 변경, 계획, diff에 범위 판단이 필요할 때 쓴다: 무엇이 필요한지, 무엇이 추측성인지, 어떤 직접 후속 수정이 필요한지, 어떻게 검증할 수 있는지.
- 변경이 추측성 작업, 과한 정리, 불필요한 재작성으로 번질 수 있을 때 쓴다.
- 변경을 너무 좁게 잡아 필요한 후속 수정이 누락될 위험이 있을 때 쓴다.
- 모호성이 구현 선택에 영향을 줄 때 쓴다.
- 아주 작은 오타 수정, 명백한 한 줄 수정처럼 검토 비용이 위험보다 큰 경우에는 쓰지 않는다.
- 프로젝트별 빌드, 테스트, 스타일, 저장소 지침을 대체하지 않는다.

## 실행 흐름

비자명한 변경에서는 아래 틀을 작게라도 명시한다.

1. 목표와 관찰 가능한 성공 기준을 정한다.
2. 함께 바뀌어야 하는 직접 영향 범위를 확인한다: 구현, 호출부, 타입, 테스트, 문서, 설정.
3. 구현에 영향을 주는 가정이나 질문을 드러낸다.
4. 목표, 성공 기준, 직접 영향 범위로 설명되는 것만 수정한다.
5. 동작을 보호하기에 충분한 가장 작은 수준에서 검증한다.
6. 변경 범위 근거와 검증 결과를 짧게 보고한다.

일상적인 작업까지 긴 의식으로 만들지 않는다. 이 틀은 변경을 분명하게 하기 위한 것이지, 그 자체가 일이 되어서는 안 된다.

## 핵심 규칙

1. **목표와 성공 기준 고정**

- 변경이 달성해야 하는 목표를 한 문장으로 정한다.
- 코드 변경 전에 성공 기준을 정한다.
- 성공 기준은 가능한 한 관찰 가능해야 한다: 실패 재현, 테스트 통과, 타입/린트 통과, 동작 확인, 명시된 요구사항 충족.
- 목표를 만족하기 위해 필요한 직접 후속 수정까지 포함한다.
- 호출부, 타입, 테스트, 문서, 설정이 직접 영향받는다면 누락하지 않는다.

2. **변경 범위 정당화**

- 변경 범위는 “가능한 가장 작은 범위”가 아니라 “목표를 달성하기에 충분하고 납득 가능한 범위”로 잡는다.
- 모든 주요 변경은 사용자 요청, 성공 기준, 기존 코드 제약 중 하나로 설명 가능해야 한다.
- 직접 영향 범위는 작업 범위에 포함한다.
- 그대로 두면 요청된 동작, 타입/체크, 문서/설정 계약, 의미 있는 검증이 깨지는 표면을 직접 영향 범위로 본다.
- 근처에 있는 불일치, 오래된 이름, 일반적인 정리 기회만으로는 범위에 포함하지 않는다.
- 직접 관련 없는 정리, 다듬기, 개선은 변경에 섞지 않는다.
- 무관한 정리 기회는 적용하지 말고 별도로 언급한다.

3. **가정 드러내기**

- 구현에 영향을 주는 가정은 코드 변경 전에 명시한다.
- 모호성이 공개 API, 아키텍처, 데이터 손실, 보안, 의존성, 비용, 마이그레이션 경로에 영향을 주면 코드 변경 전에 질문한다.
- 사소한 로컬 모호성은 합리적인 가정을 명시하고 진행한다.
- 비대화형 실행에서는 되돌리기 쉬운 가정을 선택하고 결과에 남긴다.

4. **추측성 변경 제한**

- “나중에 필요할 수 있음”만으로 기능, 옵션, 인자, 설정, 의존성, 추상화를 추가하지 않는다.
- 요청된 목표와 직접 관련 없는 리팩터링, 포맷 변경, 주석 재작성, 이름 변경을 하지 않는다.
- 기존 죽은 코드는 현재 목표와 직접 연결되거나 검증을 막는 경우에만 제거한다.
- 이번 변경으로 새로 미사용이 된 import, 변수, 함수, 파일은 정리한다.

5. **검증 맞추기**

- 검증은 변경의 위험과 성격에 맞춘다.
- 버그 수정은 가능하면 실패 재현 또는 현재 동작을 고정하는 확인을 먼저 둔다.
- 리팩터링은 변경 전후에 유지되어야 하는 동작을 확인한다.
- 기능 변경은 성공, 실패, 관련 경계 조건을 확인한다.
- 검증을 실행할 수 없다면 이유와 다음에 실행할 가장 유용한 확인을 명시한다.

## 안티패턴

- 변경을 작게 보이게 하려고 필요한 호출부, 타입, 테스트 수정을 누락하기
- 요청보다 넓은 파일이나 모듈을 함께 정리하기
- “나중에 필요할 수 있음”을 이유로 옵션, 인자, 설정, 의존성, 추상화 추가하기
- 직접적인 이유 없이 구조를 재배치하거나 이름을 바꾸기
- 기존 죽은 코드 제거를 무관한 변경에 끼워 넣기
- 검증 근거 없이 “고친 것 같다”로 마무리하기
- 실패 원인을 좁히지 않고 여러 가능성을 한 번에 바꾸기
- 주요 변경 이유를 설명할 수 없는 diff를 남기기

## 최종 확인

- 변경 목표가 분명한가?
- 성공 기준이 변경을 평가할 수 있을 만큼 분명한가?
- 필요한 직접 후속 수정을 누락하지 않았는가?
- 모든 주요 변경이 목표, 성공 기준, 기존 코드 제약 중 하나로 설명 가능한가?
- 요청과 직접 관련 없는 정리나 개선을 분리했는가?
- 추측성 기능, 옵션, 설정, 의존성, 추상화를 피했는가?
- 변경의 성격에 맞는 검증을 실행했거나, 실행하지 못한 이유를 명시했는가?

## 완료 근거

형식이 실제로 유용할 때만 아래 항목을 제시한다.

- `Goal:` 변경 목표
- `Success Criteria:` 관찰 가능한 성공 기준
- `Scope Rationale:` 변경 범위가 필요하고 납득 가능한 이유
- `Assumptions:` 구현에 영향을 주는 가정
- `Verification:` 실행/실패/생략/유예한 확인과 이유
- `Deferred:` 이번 변경에 섞지 않은 관련 정리나 후속 작업
